PROJECT: L-CDEA
MODULE: discourse/multi_hop_reasoning/

PURPOSE:
Enable deterministic multi-hop traversal over DiscourseState relationship edges.

Current behavior:
  "what does force depend on"
  → direct one-hop lookup:
     force → acceleration
     force → mass

Desired behavior:
  "what does force ultimately depend on"
  → multi-hop traversal:
     force → acceleration → velocity
     force → mass

This module moves L-CDEA from relationship retrieval to graph reasoning.

# CORE OPERATOR

GET_RELATIONSHIP_CLOSURE(term, relation_type, max_depth) -> Entity

Example:

GET_RELATIONSHIP_CLOSURE("force", "depends_on", 3)

Returns:
  direct and indirect dependency targets with paths.

---

# TRAVERSAL RULES

traversal.py:

compute_closure(term, relation_type, discourse_state, max_depth)

Rules:
1. Use breadth-first search.
2. Traverse only edges matching relation_type.
3. Include direct and indirect targets.
4. Prevent cycles with visited set.
5. Respect max_depth.
6. Sort output deterministically.

---

# RESULT TYPES

RelationshipPath {
  source: str
  relation_type: str
  target: str
  depth: int
  path: Tuple[str, ...]
  provenance: Tuple[Provenance, ...]
}

RelationshipClosureResult {
  source_term: str
  relation_type: str
  max_depth: int
  paths: Tuple[RelationshipPath, ...]
  fallback_used: bool
}

---

# SORTING RULES

Paths must be sorted by:

1. depth ascending
2. target value lexicographic
3. path string lexicographic

Example order:

force → acceleration
force → mass
force → acceleration → velocity

---

# PROVENANCE RULES

Each path MUST aggregate edge provenance.

Path provenance:

force → acceleration → velocity

Includes:
1. provenance from force → acceleration edge
2. provenance from acceleration → velocity edge

Display:
- per path
- show strongest provenance entry first

---

# CYCLE HANDLING

If graph contains:

A → B
B → A

Traversal MUST:
1. avoid infinite loop
2. record cycle_detected=True in trace
3. return valid acyclic paths only

---

# OPERATOR BEHAVIOR

GET_RELATIONSHIP_CLOSURE executes by:

1. normalizing term
2. normalizing relation_type
3. parsing max_depth
4. calling compute_closure
5. returning RelationshipClosureResult

If no paths found:

return:
  closure_of(term, relation_type)

fallback_used=True

---

# PLANNER INTEGRATION

Planner MUST support:

domain = "discourse"
operator = "GET_RELATIONSHIP_CLOSURE"

Behavior:
1. hydrate term as Entity
2. hydrate relation_type as Entity
3. hydrate max_depth as Entity or Constraint
4. short-circuit closure lookup before graph build when possible
5. cache only real hits
6. fallback is NOT cache_hit

# HARD RULES

1. Multi-hop reasoning MUST be deterministic.
2. It MUST NOT modify DiscourseState.
3. It MUST NOT modify CDL semantics.
4. It MUST avoid infinite loops.
5. It MUST respect max_depth.
6. It MUST preserve provenance.
7. It MUST be trace-visible.
8. It MUST not use external services.
9. It MUST not bypass governance.
10. Fallback must not be cache_hit.
