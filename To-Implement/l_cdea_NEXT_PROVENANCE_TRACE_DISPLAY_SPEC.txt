PROJECT: L-CDEA
MODULE: trace/provenance_display/

PURPOSE:
Surface provenance information for retrieved knowledge directly in query output and trace logs.

Current state:
- Provenance is correctly stored in DiscourseState
- Definition retrieval returns correct values
- Trace does NOT display:
    source_path
    chunk_id
    location
    extraction_method
    confidence

This limits observability and debugging.

Desired behavior:
Every retrieved value should clearly show where it came from.

---

# POSITION IN PIPELINE

Query
→ Router
→ Planner
→ Retrieval / Execution
→ Provenance Lookup (THIS MODULE)
→ Output + Trace

# CORE TYPES

DisplayedProvenance {
  source_path: Optional[str]
  chunk_id: Optional[str]
  location: Optional[str]
  extraction_method: str
  confidence: float
  trace_id: str
}

---

# PROVENANCE EXTRACTION

extractor.py:

extract_provenance(node_or_value) -> Tuple[DisplayedProvenance, ...]

Rules:

1. Pull provenance entries from:
   - DiscourseNode.provenance
   - ProvenancedValue.provenance

2. Convert to DisplayedProvenance

3. Preserve ordering:
   - highest confidence first
   - deterministic tie-break

4. Deduplicate by trace_id

---

# OUTPUT FORMATTER

formatter.py:

format_provenance(entries) -> str

Default format:

[source: <file> | chunk: <id> | loc: <location> | conf: <confidence>]

Example:

Velocity is displacement over time.
  [source: physics_terms.txt | chunk: chunk_9f82a1c3 | loc: line:1 | conf: 0.92]

Multiple entries:

Acceleration is the rate of change of velocity.
  [source: physics_terms.txt | chunk: chunk_a1b2c3d4 | loc: line:3 | conf: 0.91]
  [source: textbook_v1.txt | chunk: chunk_ff91aa22 | loc: page:12 | conf: 0.87]

---

# TRACE INTEGRATION

Modify trace output:

Add section:

[PROVENANCE]

For each returned value:

term: velocity
entries:
  - source_path: physics_terms.txt
    chunk_id: chunk_9f82a1c3
    location: line:1
    extraction_method: definition_extractor
    confidence: 0.92
    trace_id: ...

# CLI OUTPUT CHANGE

Rules:

# PRIORITY RULES

When multiple provenance entries exist:

Sort by:
1. confidence (desc)
2. salience (if available)
3. timestamp_index (desc)
4. deterministic tie-break

Limit to max_entries

---

# HARD RULES

1. Provenance display MUST be deterministic.
2. Must NOT modify stored provenance.
3. Must NOT affect execution or retrieval logic.
4. Must work for:
   - definition retrieval
   - dataset lookup
   - execution results
5. Must NOT break existing trace format.
6. Must degrade gracefully when provenance missing.
