PROJECT: L-CDEA
MODULE: trace/

PURPOSE:
Provide full deterministic observability for the complete L-CDEA pipeline.

The trace logger records every major decision and transformation from:

parse
→ router
→ planner
→ compile
→ canonicalize
→ MECP
→ execute
→ discourse

This module exists so failures, cache hits, operator choices, slot ordering, graph construction, and discourse updates are inspectable and reproducible.

---

# PIPELINE POSITION

Input
→ Parser
→ Router
→ Planner
→ Compiler / Direct Graph
→ CAS
→ MECP
→ Execution
→ DiscourseState
→ Trace Logger

Trace Logger observes the pipeline.
It MUST NOT affect pipeline behavior.

# HARD RULES

1. Trace logger MUST be deterministic.
2. Trace logger MUST NOT mutate pipeline state.
3. Trace logger MUST NOT influence MECP, planner, router, compiler, execution, or discourse.
4. Trace logger MUST preserve stage ordering.
5. Trace logger MUST include enough data to replay or debug a result.
6. Trace logger MUST record errors without crashing the system.
7. Trace logger MUST not use random IDs unless seeded/deterministic.
8. Trace logger MUST not call external services in V1.

---

# DETERMINISTIC ID RULE

Trace IDs and event IDs MUST be deterministic in V1.

Recommended format:

trace_id =
  stable hash of input_text + pipeline_version + discourse_snapshot_id

event_id =
  trace_id + stage + timestamp_index

Do NOT use uuid4() in V1.

---

# SINKS

V1 sinks:

1. InMemoryTraceSink
2. JSONLTraceSink

Rules:

InMemoryTraceSink:
- stores trace records in memory for tests

JSONLTraceSink:
- writes one TraceRecord per line
- append-only
- deterministic serialization order

# FORMATTER

formatter.py MUST provide:

to_dict(record)
to_json(record)
to_pretty_text(record)

Pretty output should be human-readable and include:

- input
- selected intent
- cache hit/miss
- selected operator
- MECP selected graphs
- execution output
- discourse update

---

# REPLAY SUPPORT

replay.py MUST provide minimal replay metadata.

V1 replay does not need to re-run the full pipeline.

It MUST allow inspection of:

1. original input
2. stage sequence
3. selected intent
4. selected operator
5. selected execution graph
6. final output

---

# ERROR HANDLING

If any stage fails:

Trace logger MUST record:

stage
error_type
error_message
payload_available

Then final_status = "failed"

If pipeline succeeds:

final_status = "success"

# VALIDATION EXAMPLES

## Example 1 — Router + Planner Trace

Input:
  "calculate acceleration from force 10 N and mass 2 kg"

Expected trace includes:

ROUTER:
  selected_intent.domain = physics
  selected_intent.operator_name = COMPUTE_ACCELERATION
  arg_order = ("force", "mass")

PLANNER:
  hydrated_slots = {
    force: {"value": 10, "unit": "N"},
    mass: {"value": 2, "unit": "kg"}
  }
  slot_order_used = ("force", "mass")
  retyped_slots use operator signature order
  graph_built = true

EXECUTION:
  output = {"value": 5.0, "unit": "m/s²"}

# DESIGN INTENT

Trace is the observability layer.

It exists to answer:

1. What did the system parse?
2. Which domain/operator did it route to?
3. Was the answer found in memory?
4. How were slots hydrated and ordered?
5. Which graph was built?
6. What did MECP select?
7. What executed?
8. What entered DiscourseState?

Trace MUST make the system debuggable without changing behavior.
