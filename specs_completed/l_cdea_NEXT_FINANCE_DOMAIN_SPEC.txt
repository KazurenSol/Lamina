PROJECT: L-CDEA
MODULE: domain/finance/

PURPOSE:
Provide a high-stress domain pack for modeling financial systems, transactions, time-dependent value, and risk/return tradeoffs.

---

# DOMAIN ROLE

Finance Domain V1 is designed to stress:

1. Time-dependent transformations (interest, growth, decay)
2. Numerical + logical reasoning combined
3. MECP tradeoff decisions (risk vs return vs cost)
4. DiscourseState accumulation of transactional history
5. Multi-step computation pipelines
6. Integration with math + programming domains

This domain introduces **temporal reasoning + accumulation**, which is critical for real-world applications.

# FINANCE OPERATORS V1

# Core transactions

DEPOSIT(account, amount) -> Balance
WITHDRAW(account, amount) -> Balance
TRANSFER(source, target, amount) -> Balance

# Value computation

COMPUTE_INTEREST(principal, rate, time) -> Money
COMPUTE_COMPOUND_INTEREST(principal, rate, time, frequency) -> Money
UPDATE_BALANCE(account, transaction) -> Balance

# Investment

INVEST(asset, amount) -> Investment
CALCULATE_RETURN(investment, time) -> Return
EVALUATE_RISK(asset) -> Risk

# Portfolio

ADD_ASSET(portfolio, asset) -> Portfolio
REMOVE_ASSET(portfolio, asset) -> Portfolio
COMPUTE_PORTFOLIO_VALUE(portfolio) -> Money

# Time operations

ADVANCE_TIME(state, time_delta) -> State

# Comparison

COMPARE_VALUES(value1, value2) -> Constraint

---

# REQUIRED CORE OPERATOR MAPPINGS

DEPOSIT
= TRANSFORM(balance + amount)

WITHDRAW
= TRANSFORM(balance - amount)

TRANSFER
= TRANSFORM(source - amount, target + amount)

COMPUTE_INTEREST
= TRANSFORM(principal * rate * time)

COMPUTE_COMPOUND_INTEREST
= TRANSFORM(principal * (1 + rate)^time)

CALCULATE_RETURN
= TRANSFORM(final_value - initial_value)

EVALUATE_RISK
= CLASSIFY(asset risk level)

# TEMPORAL RULES

Finance requires explicit time modeling.

RULES:

1. All growth/decay MUST include time input
2. No implicit time progression
3. Time must be tracked in DiscourseState
4. MECP may favor faster analytical solutions over simulation

---

# CANONICALIZATION RULES

CAS MUST detect equivalent financial expressions.

Examples:

Simple interest:
  A = P * (1 + r * t)

Compound interest:
  A = P * (1 + r)^t

Equivalent representations MUST normalize when mathematically identical.

Transaction sequences that result in same balance MUST be equivalent.

# EXECUTION RULES

1. All finance operators MUST be deterministic
2. No randomness in V1
3. No external financial APIs in V1
4. All computations MUST be reproducible

---

# DISCOURSESTATE STORAGE RULES

Finance memory should track:

- accounts and balances
- transaction history
- investments
- time evolution of value

Example:

Account:
  balance = 1000

Transaction:
  deposit 200

Result:
  balance = 1200
